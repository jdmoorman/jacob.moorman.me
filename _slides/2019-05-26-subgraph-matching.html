--- layout: presentation title: Subgraph Matching (ATC) order: 9 --- ## Subgraph Matching **Jacob Moorman** -vertical- In collaboration with *Thomas Tu, Qinyi Chen, Xie He, and Andrea Bertozzi* of UCLA Department of Mathematics -vertical- Work funded by DARPA under the Modeling Adversarial Activity (MAA) program -vertical- Jacob was funded by NSF grant DGE-1829071 -vertical- Slides are online at [bit.ly/jacobsm](http://bit.ly/jacobsm) Code is on GitHub at [bit.ly/ucla-sm](http://bit.ly/ucla-sm) -horizontal- ## Subgraph Matching Find all copies of a **template** network in a **world** network An instance of the *template* in the *world* is called a **signal** -vertical- ## Backstory 1. DARPA hired several teams (PNNL, IvySys, GORDIAN) to generate synthetic networks 2. DARPA challenged other teams to create algorithms to solve subgraph matching on those networks 3. We succeeded. -vertical- ## An easier problem | Template Node | Signal Nodes | |:---:|:---:| | A | 1, 4, 7 | | B | 2, 5, 8, 9 | | C | 6, 7, 10 | -horizontal- ## Classic Algorithm: ## Tree Search [Ullmann76, CFSV04] -vertical- 1. Find all signals where \\(\mathbf{A} = \mathbf{1}\\) 1. Find all signals where \\(\mathbf{A} = \mathbf{1}\\) and \\(\mathbf{C} = \mathbf{6}\\) 1. ... 2. Find all signals where \\(\mathbf{A} = \mathbf{4}\\) 3. ... -vertical- ## Tree Search Considerations * **Search Order:** In what order should we match the template nodes? * **Filtering Rules:** To which world nodes should we try matching them? -horizontal- ## Search Order > *In what order should we match the template nodes?* Use a heuristic to precompute a fixed order on the template nodes. -vertical- ## Search Order Objective The objective of the heuristic is to minimize the computational cost of traversing the search tree. -vertical- ## Search Order Heuristics * Fewest candidates first * Most neighbors first * Highest degree first * ... -horizontal- ## Filtering Rules > To which world nodes should we try matching the template nodes? Try matching each template node to world nodes which satisfy some set of constraints. Refer to these world nodes as **candidates**. -vertical- ## Filtering Rules: Node Stats A candidate should have as many incoming/outgoing edges as its corresponding template node. -vertical- | Template Node | Candidates After Stats| |:---:|:---:| | A | 1, 3, 4 | | B | 1, 2, 3, 4 | | C | 4 | -vertical- ## Filtering Rules: Topology Candidates for neighboring template nodes should neighbor other candidates for the corresponding template nodes. -vertical- | Template Node | Candidates Before Topology | Candidates After Topology | |:---:|:---:|:---:| | A | 1, 3, 4 | 1 | | B | 1, 2, 3, 4 | 1, 2 | | C | 4 | 4 | -vertical- ## Filtering Rules: Pigeonhole If **k** world nodes are candidates for the same **k** template nodes, they cannot be candidates for any others. -vertical- | Template Node | Candidates Before Pigeonhole | Candidates After Pigeonhole | |:---:|:---:|:---:| | A | 1 | 1 | | B | 1, 2 | 2 | | C | 4 | 4 | -vertical- ## Filtering Rules: Neighborhood [Solnon09] The neighborhood of a candidate should be matchable to the neighborhood of the corresponding template node. -horizontal- ## Results Filtering rules alone uniquely identify many nodes! -vertical- Expensive filtering rules are not always worth the effort! -vertical- Using a tree search, we are usually able to verify exactly which candidates belong to at least one signal. We are not always able to enumerate all of the signals. -horizontal- ## Future Work * Approximate matching in the presence of noise * Missing nodes/edges * Incorrect attribute info * Attributes beyond node/edge labels (eg. timestamps) -vertical- ## References
